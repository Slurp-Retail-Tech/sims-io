### SIMS-01 — Critical: Majority of API routes authenticate via forgeable `x-user-id` header

**Description**

40 out of 45 protected API route files use the following pattern to identify the caller:

```ts
const userId = request.headers.get("x-user-id")?.trim()
if (!userId) {
  return NextResponse.json({ error: "Unauthorized." }, { status: 401 })
}
```

Any HTTP client (browser, curl, Postman) can send an arbitrary `x-user-id` header. The middleware only checks for the presence of the `sims-auth` cookie on page routes — it does not cover API routes. An unauthenticated attacker can bypass this check entirely by simply including the header.

The only routes that correctly use `requireAuthenticatedUser` (which reads the httpOnly cookie and validates it against the database) are: `/api/auth/session`, `/api/profile`, `/api/users`, `/api/users/[userId]`, and `/api/users/agents`.

**Affected routes (representative, not exhaustive)**
- `src/app/api/tickets/route.ts`
- `src/app/api/tickets/[ticketId]/route.ts`
- `src/app/api/merchants/**`
- `src/app/api/leads/route.ts` (GET — listing leads)
- `src/app/api/leads/[leadId]/route.ts`
- `src/app/api/plus/**`
- `src/app/api/clickup/**`
- `src/app/api/renewals/expiring/route.ts`
- `src/app/api/maps/outlets/route.ts`
- `src/app/api/onboarding-appointments/**` (via `helpers.ts:88`)
- `src/app/api/clickup-task-requests/**` (via `helpers.ts:95`)
- `src/app/api/uploads/route.ts`

**Risk**

An unauthenticated external actor (or any user who can issue HTTP requests) can:
- Read all tickets, merchant data, leads, and outlet maps
- Archive or modify leads
- Trigger merchant and PLUS data imports
- Upload arbitrary files to MinIO
- Create, approve, or reject ClickUp task requests
- Share CSAT links

**Recommendation**

Replace all `request.headers.get("x-user-id")` checks with `requireAuthenticatedUser(request)` from `src/lib/auth.ts`. The `x-user-id` header should be removed from all client-side fetch calls once the server reads identity from the cookie. The helper `resolveAuthUser` in `onboarding-appointments/helpers.ts` and `clickup-task-requests/helpers.ts` should be refactored to call `requireAuthenticatedUser` internally.

**Verification**

```bash
curl -s -X GET "http://localhost:3000/api/tickets" -H "x-user-id: 1"
# Should return 401 — currently returns ticket data
```

### SIMS-02 — Critical: Session cookie stores raw user ID, not an opaque session token

**Description**

`setAuthCookie` stores `user.id` (a database integer) directly in the `sims-auth` cookie:

```ts
// src/lib/auth.ts:136
response.cookies.set(SESSION_COOKIE_NAME, userId, { ... })
```

`getAuthenticatedUser` reads this value and queries the `users` table directly:

```ts
const userId = request.cookies.get(SESSION_COOKIE_NAME)?.value?.trim()
// ...
WHERE id = ?
```

There is no `sessions` table referenced anywhere in `schema.sql` or the codebase, despite CLAUDE.md describing an architecture with hashed session tokens.

**Risk**

- The cookie value is a predictable, sequential integer (e.g. `1`, `2`, `3`). Brute-forcing or guessing the value of any user's session is trivial.
- There is no server-side session invalidation. After a user changes their password, existing cookies for other sessions remain valid indefinitely. A stolen cookie cannot be revoked without deactivating the account.
- Logout only clears the client-side cookie (`clearAuthCookie`); if the cookie value is known to an attacker, they continue to have access.

**Recommendation**

1. Create a `sessions` table with columns `(id, user_id, token_hash CHAR(64), expires_at, created_at)`.
2. On login, generate a 32-byte random token via `createOpaqueToken()`, store its SHA-256 hash in `sessions`, and write the raw token to the cookie.
3. In `getAuthenticatedUser`, hash the cookie value and JOIN against `sessions`.
4. On logout and password reset, `DELETE FROM sessions WHERE token_hash = ?` (or `WHERE user_id = ?` for all sessions).

### SIMS-03 — High: `/api/uploads/view` serves arbitrary MinIO objects with no authentication

**Description**

```ts
// src/app/api/uploads/view/route.ts:6
export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url)
  const key = searchParams.get("key")?.trim()
  // No auth check whatsoever
  const result = await getObjectStream(bucket, key)
  return new NextResponse(stream, ...)
}
```

Any caller who knows (or guesses) a storage key can retrieve the corresponding file from MinIO. Object keys follow a deterministic format: `{folder}/{userId}/{timestamp}-{random}.{ext}`. The timestamp component provides limited entropy. There is no validation that the requesting user has any relationship to the object being retrieved.

**Risk**

- Support form attachments (bug reports, receipts) are stored in `support-form/public/` — the `userId` segment is the literal string `"public"` — making the path prefix entirely predictable.
- Avatar images and ticket attachments are exposed to any unauthenticated party who constructs or enumerates object keys.
- Content-Type is reflected directly from MinIO metadata, meaning an attacker could upload and then serve arbitrary HTML, SVG, or JavaScript files through this endpoint.

**Recommendation**

1. Require authentication on this endpoint using `requireAuthenticatedUser`.
2. For truly public content (support form attachments that are shared with merchants), consider generating short-lived pre-signed S3 URLs instead of a permanent proxy.
3. Validate that the key prefix corresponds to a bucket+path the requesting user is authorised to read (e.g. `avatars/{user.id}/` for avatar access).

### SIMS-04 — High: No rate limiting on any auth or public endpoint

**Description**

CLAUDE.md documents a rate limiting system (`src/lib/rate-limit.ts`, `src/lib/rate-limit-store.ts`) with specific limits: login (10/15 min), forgot-password (5/15 min), support form (5/min), leads form (3/min). Neither of these files exists in the codebase. No rate limiting of any kind is applied to any API route.

Affected endpoints:
- `POST /api/auth/login` — unlimited password guessing
- `POST /api/auth/forgot-password` — unlimited email enumeration and email bombing
- `POST /api/supportform/submit` — unlimited ticket creation and file uploads
- `POST /api/leads` (public) — unlimited lead spam (reCAPTCHA is optional, see SIMS-10)
- `POST /api/csat/[token]` — unlimited CSAT response submissions

**Risk**

- Credential stuffing and brute-force attacks against all user accounts are unrestricted.
- A single attacker can exhaust SMTP quotas via repeated forgot-password requests.
- Public endpoints can be used to flood the database with spam records.

**Recommendation**

Implement rate limiting using a Redis-backed counter (sliding window). Middleware or a shared utility should enforce limits before business logic executes. Key on a combination of IP address and endpoint. Use the `x-forwarded-for` header only when `TRUSTED_PROXY` is set (see also SIMS-09).

### SIMS-05 — High: CSAT tokens stored as plaintext in database

**Description**

`auth_tokens` (activation and password reset) correctly stores only a SHA-256 hash (`token_hash CHAR(64)`). However, `csat_tokens` stores the raw token value:

```sql
-- schema.sql:277
token VARCHAR(255) NOT NULL UNIQUE,
```

The token is generated with `randomUUID()` and inserted without hashing:

```ts
// src/app/api/tickets/[ticketId]/route.ts:614
generatedCsatToken = randomUUID()
await pool.query(`INSERT INTO csat_tokens (ticket_id, token, ...) VALUES (?, ?, ...)`, [ticketId, generatedCsatToken])
```

Lookups in `csat/[token]/route.ts` also query by plaintext `WHERE csat_tokens.token = ?`.

**Risk**

A database read (via SQL injection, a compromised DB admin account, or a backup/log leak) exposes all active CSAT survey links. An attacker can submit fraudulent CSAT responses on behalf of any merchant for any open ticket, manipulating satisfaction metrics.

**Recommendation**

Migrate `csat_tokens` to store `token_hash CHAR(64)` (SHA-256 of the raw token, matching the `auth_tokens` pattern). Send the raw token in the survey URL, hash it on lookup. Apply a migration that re-hashes any existing tokens.

### SIMS-06 — High: `xlsx` 0.18.5 has known high-severity CVEs and parses attacker-uploaded files

**Description**

`package.json` pins `xlsx` at `^0.18.5`, which has two active high-severity advisories:
- GHSA-4r6h-8v6p-xvw6
- GHSA-5pgg-2g8v-p4x9

This library is used to parse user-controlled XLSX files in the PLUS update flow:

```ts
// src/lib/plus-import.ts:173
const workbook = XLSX.read(buffer, { type: "buffer" })
```

The buffer comes from a file uploaded by an authenticated user via `/api/uploads`, fetched from MinIO, then passed directly to `XLSX.read`.

**Risk**

Parsing a maliciously crafted XLSX file could trigger prototype pollution or denial-of-service via the known vulnerabilities. Any authenticated user with access to the PLUS page can trigger this parse path.

**Note on exploitability**: The `xlsx` package has been effectively abandoned by its maintainer. The advisories on `GHSA-4r6h-8v6p-xvw6` relate to ReDoS/prototype pollution. `npm audit` confirms these are present in the installed version.

**Recommendation**

Migrate to a maintained alternative such as `exceljs` or `@e965/xlsx`. Validate that uploaded files do not exceed expected row counts and column structures before parsing. Restrict PLUS page access to `Super Admin` / `Admin` roles only if not already enforced.

### SIMS-07 — High: HTTP security headers entirely absent from `next.config.ts`

**Description**

`next.config.ts` contains no `headers()` configuration. None of the following headers are set:

- `Content-Security-Policy` — no protection against XSS injection
- `X-Frame-Options` / `frame-ancestors` CSP directive — clickjacking possible
- `X-Content-Type-Options: nosniff` — MIME sniffing attacks possible
- `Referrer-Policy` — sensitive URL data (reset tokens in URLs) may leak via `Referer` header to third parties
- `Strict-Transport-Security` — HSTS not enforced
- `Permissions-Policy` — no restrictions on camera, geolocation, etc.

**Risk**

The absence of `Content-Security-Policy` means that any XSS vulnerability (including those introduced by SIMS-13) can exfiltrate session cookies or make authenticated API calls. The absence of `X-Frame-Options` allows the application to be embedded in an iframe for clickjacking attacks. Without `Referrer-Policy: no-referrer`, password reset tokens that appear in reset URLs could leak to external services (analytics, CDN logs) via the `Referer` header.

**Recommendation**

Add a `headers()` export to `next.config.ts`. A minimal starting configuration:

```ts
async headers() {
  return [
    {
      source: "/(.*)",
      headers: [
        { key: "X-Frame-Options", value: "DENY" },
        { key: "X-Content-Type-Options", value: "nosniff" },
        { key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
        { key: "Strict-Transport-Security", value: "max-age=63072000; includeSubDomains" },
        { key: "Permissions-Policy", value: "camera=(), microphone=(), geolocation=()" },
      ],
    },
  ]
}
```

A Content-Security-Policy should be defined once the `dangerouslySetInnerHTML` usage (SIMS-13) is resolved.

### SIMS-08 — Medium: Inconsistent password minimum length across routes

**Description**

Password length enforcement differs across the three paths where a password can be set:

| Route | Minimum Length |
|-------|----------------|
| `POST /api/auth/activate` | 6 characters (`if (password.length < 6)`) |
| `POST /api/auth/reset-password` | None |
| `PATCH /api/users/[userId]` (admin sets password) | None |
| `PATCH /api/profile` (user changes own password) | None |

CLAUDE.md documents the minimum as 12 characters.

**Risk**

A password reset or admin-assigned password can be set to a single character. This bypasses any intended security policy and makes accounts easier to brute-force.

**Recommendation**

Enforce a consistent minimum of 12 characters (per CLAUDE.md) in all four paths: activate, reset-password, profile PATCH, and users/[userId] PATCH. Move the check into a shared `validatePassword(password: string)` function in `src/lib/auth.ts`.

### SIMS-09 — Medium: `x-forwarded-for` read directly without `TRUSTED_PROXY` guard

**Description**

The leads route passes the raw `x-forwarded-for` header to Google's reCAPTCHA verification without checking whether the proxy is trusted:

```ts
// src/app/api/leads/route.ts:353
const verified = await verifyRecaptchaToken(
  recaptchaToken,
  request.headers.get("x-forwarded-for")  // unguarded
)
```

CLAUDE.md documents that `x-forwarded-for` should only be read when `TRUSTED_PROXY` is set. The leads route ignores this constraint.

**Risk**

An attacker can set a spoofed `x-forwarded-for` header to provide a trusted IP address to the reCAPTCHA `remoteip` parameter, potentially bypassing IP-based risk scoring in reCAPTCHA v3.

**Recommendation**

Apply the same `TRUSTED_PROXY` guard that CLAUDE.md describes for rate limiting: only read `x-forwarded-for` when `process.env.TRUSTED_PROXY` is set, otherwise pass `null` (or the direct connection address).

### SIMS-10 — Medium: reCAPTCHA enforcement on public lead form is optional

**Description**

reCAPTCHA verification is gated entirely on whether `RECAPTCHA_SECRET_KEY` is configured:

```ts
// src/app/api/leads/route.ts:346
if (process.env.RECAPTCHA_SECRET_KEY?.trim()) {
  // enforce reCAPTCHA
}
// If not configured, all submissions pass through
```

If the environment variable is not set (e.g., in staging, or if accidentally removed from production), the public `POST /api/leads` endpoint accepts unlimited submissions with no bot protection.

**Risk**

The public lead form can be spammed to exhaustion, flooding the leads table, triggering HubSpot API calls for each submission, and sending bulk email notifications to recipients configured in `lead_notification_settings`.

**Recommendation**

Either fail closed (reject submissions when reCAPTCHA is configured but verification fails, and require the key to be set in production) or add an explicit `if (process.env.NODE_ENV === 'production' && !secret) { reject }` guard. Document clearly that `RECAPTCHA_SECRET_KEY` is required in production.

### SIMS-11 — Medium: Default admin account seeded with a static password hash in `schema.sql`

**Description**

`schema.sql` line 369–379 inserts a `Super Admin` user with a hardcoded `password_hash`. This hash is committed to the repository and visible to anyone with repository access.

```sql
INSERT INTO users (name, email, department, role, status, is_active, password_hash)
VALUES (
  'Super Admin',
  'admin@getslurp.com',
  ...
  'e910ed24ba65bf62f83ade706cd2d378:d7f8f1e86d39ac661c5bba0ef1c3337c...'
)
```

The hash uses the application's scrypt format (`salt:hash`). Any developer with repository access can attempt to crack this hash offline, or — if the same password is used in production — impersonate the account.

**Risk**

If the production database was seeded from this SQL file and the default password has not been changed, the Super Admin account is compromised to anyone with Git history access. Even if changed, the email `admin@getslurp.com` is a known target.

**Recommendation**

Remove the `password_hash` from `schema.sql`. Replace with a sentinel that forces activation on first login (set `status = 'pending_activation'`, `password_hash = NULL`). Add a note in `README.md` requiring the admin account password to be set immediately after first deployment via the activation flow.

### SIMS-12 — Medium: Session and state cookies use `SameSite=lax` instead of documented `strict`

**Description**

CLAUDE.md documents session cookies as `SameSite=strict`. The actual implementation uses `lax` for both the session cookie and the Google OAuth state cookie:

```ts
// src/lib/auth.ts:138,148
sameSite: "lax"

// src/app/api/auth/google/start/route.ts:33
sameSite: "lax"
```

`SameSite=lax` allows the cookie to be sent on top-level cross-site navigations (i.e., clicking a link to the app from an external site). `SameSite=strict` would prevent this.

**Risk**

For the OAuth state cookie (`sims-google-state`), `lax` is actually the correct setting — `strict` would break the OAuth callback flow (the redirect from Google back to the app would not include the state cookie). The documentation is incorrect for this cookie.

For the session cookie (`sims-auth`), `lax` combined with the CSRF posture (no explicit CSRF tokens) means that cross-site top-level navigations do send the session cookie. Since all state-changing operations in the app are JSON POST/PATCH requests (not form submissions), `lax` provides equivalent CSRF protection in practice. However, this diverges from documentation and should be reconciled.

**Recommendation**

Update CLAUDE.md to reflect `SameSite=lax` with an explanation of why `strict` is not used (OAuth callback compatibility). Alternatively, use `strict` for the session cookie only, and document that the OAuth state cookie must remain `lax`.

### SIMS-13 — Medium: `dangerouslySetInnerHTML` injects chart config values without sanitisation

**Description**

`ChartStyle` in `src/components/ui/chart.tsx` builds a CSS `<style>` block using `dangerouslySetInnerHTML`. The `cfg.color` and `key` values from `ChartConfig` are interpolated without escaping:

```tsx
// src/components/ui/chart.tsx:43
return `[data-chart=${id}] { --color-${key}: ${cfg.color}; }`
```

The `id` value comes from `chartId = \`chart-${id ?? uniqueId}\`` where `id` is a prop. The `config` prop is the `ChartConfig` object.

**Current exploitability: Low.** Chart configs in the codebase (`trendConfig`, `sourcesConfig`, etc.) are hardcoded static constants with hex colour values. No user-supplied data flows into these config objects today.

**Risk**

If chart configurations are ever driven by user data (e.g., custom category names used as config keys, or user-defined colours), an attacker could inject arbitrary CSS or break out of the style block. CSS injection in a `<style>` tag can be used for data exfiltration via CSS selectors in some browser environments.

**Recommendation**

Add CSS value validation: assert that `cfg.color` matches a safe colour pattern (`/^#[0-9a-fA-F]{3,8}$|^rgb\(|^hsl\(/`) before interpolation, and that `key` matches `/^[a-z0-9_-]+$/i`. Alternatively, use `document.documentElement.style.setProperty()` via a `useEffect` to set CSS variables programmatically, avoiding `dangerouslySetInnerHTML` entirely.

### SIMS-14 — Low: `NEXT_PUBLIC_APP_URL` influences server-side redirect base URL

**Description**

`resolveAppBaseUrl` in `src/lib/auth.ts:205` uses `NEXT_PUBLIC_APP_URL` as a fallback:

```ts
const configured =
  process.env.APP_BASE_URL?.trim() ||
  process.env.NEXT_PUBLIC_APP_URL?.trim() ||
  ""
```

`NEXT_PUBLIC_*` variables are baked into the client-side JavaScript bundle at build time. They are not secret and should not be used for security-sensitive server-side decisions. If the build environment is compromised or the env var is misconfigured, this value could produce unintended redirect targets.

**Risk**

Low in practice — `APP_BASE_URL` takes precedence and is the documented production variable. The risk is primarily a misconfiguration hazard.

**Recommendation**

Remove `NEXT_PUBLIC_APP_URL` from the fallback chain in `resolveAppBaseUrl`. Only `APP_BASE_URL` (a server-only variable) should determine redirect targets.

### SIMS-17 — Info: Client-side `localStorage` session cache is the source for `x-user-id`

**Description**

`src/lib/session.ts` caches the full `SessionUser` object (including `id`) in `localStorage`. Client-side components read `user.id` from this cache and pass it as `x-user-id` to API calls.

This is the mechanism by which `x-user-id` is populated. It is not an independent vulnerability but the root cause of why SIMS-01 exists: the client informs the server of its own identity rather than the server inferring identity from an httpOnly cookie.

**Assessment**

Resolving SIMS-01 (replacing `x-user-id` with `requireAuthenticatedUser`) will make the `localStorage` session cache a pure UI convenience (display name, role for client-side rendering) rather than a security boundary. That is an acceptable use of `localStorage`. No separate fix required beyond SIMS-01.
