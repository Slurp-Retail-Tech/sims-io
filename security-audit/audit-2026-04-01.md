# SIMS Security Audit Report

**Date:** 2026-04-01
**Auditor:** Claude Code (Anthropic)
**Scope:** Full codebase audit — frontend and backend
**Branch audited:** staging

---

## Executive Summary

SIMS is a Next.js App Router application managing merchant engagement, support ticketing, leads, renewals, and appointments. The codebase is generally well-structured, uses parameterized queries throughout, and avoids the most obvious injection vulnerabilities. However, the audit identified one critical architectural flaw and several high-severity issues that could allow unauthenticated or low-privilege actors to access or manipulate internal data.

The most severe finding is that the dominant authentication mechanism for API routes is an HTTP header (`x-user-id`) that is set by the browser client and never cryptographically verified server-side. Any browser session that has read the session user ID (which is exposed via `/api/auth/session`) can forge requests to any route protected only by this header. The correct session cookie is never validated for the majority of the application's API surface. A secondary high-severity finding is that the file proxy endpoint (`/api/uploads/view`) is completely unauthenticated, exposing all stored files to anyone who knows or can guess an object key.

Other significant findings include: no rate limiting on any authentication or public-facing endpoint, weak minimum password policy (6 characters), missing security response headers, a plain-HTTP default for a POS API import URL, and several high-severity dependency vulnerabilities.

## Architecture and Attack Surface Overview

### Route Structure

| Path Prefix | Auth Model | Notes |
|---|---|---|
| `/api/auth/*` | None / token-based | Login, Google SSO, password reset, activation |
| `/api/supportform/*` | None | Fully public ticket submission |
| `/api/leads` POST | None / reCAPTCHA | Public demo form (reCAPTCHA optional) |
| `/api/csat/*` | Opaque token only | Public CSAT submission |
| `/api/uploads/view` | **None** | Proxy for all uploaded files |
| Most other `/api/*` | `x-user-id` header | Forged-able client header |
| `/api/users/*`, `/api/profile/*` | `sims-auth` cookie | Proper session cookie |

### Authentication Flow

1. Login (`/api/auth/login`) sets an `httpOnly` cookie `sims-auth` containing the raw database user ID.
2. Middleware checks for the cookie on page routes under `(app)/` but does **not** wrap API routes — API routes must self-validate.
3. Most API routes accept a plain `x-user-id` HTTP header for "authentication" instead of verifying the `sims-auth` cookie.
4. `/api/auth/session` returns the full session user object (id, name, email, role, etc.) to any browser that has the cookie.
5. The user ID in the cookie is the literal database primary key, not a signed session token.

### Trust Boundaries

- Browser (authenticated) → API routes (should be server-verified)
- Cron/scheduler → `/api/clickup/sync` (protected by `CLICKUP_SYNC_CRON_SECRET`)
- External: Google OAuth, HubSpot, ClickUp, POS API, MinIO, SMTP

### CRITICAL

#### C-1: `x-user-id` Header Used as Authentication on ~40 API Endpoints

**Risk:** Complete authentication bypass. Any actor who can make HTTP requests to the server and knows any valid user's numeric database ID can impersonate that user across the majority of the API surface.

**Affected Files (representative):**
- `/Users/hafiz/sims-io/src/app/api/tickets/route.ts` — line 6
- `/Users/hafiz/sims-io/src/app/api/tickets/[ticketId]/route.ts` — lines 161, 318
- `/Users/hafiz/sims-io/src/app/api/merchants/export/route.ts` — line 382
- `/Users/hafiz/sims-io/src/app/api/merchants/import/route.ts` — line 6
- `/Users/hafiz/sims-io/src/app/api/plus/update/route.ts` — line 12
- `/Users/hafiz/sims-io/src/app/api/plus/cleanup/route.ts` — line 6
- `/Users/hafiz/sims-io/src/app/api/uploads/route.ts` — line 14
- `/Users/hafiz/sims-io/src/app/api/ticket-categories/route.ts` — lines 52, 72, 129, 188
- `/Users/hafiz/sims-io/src/lib/upload-client.ts` — line 24
- Full list of ~40 affected routes: all routes using the pattern `request.headers.get("x-user-id")`

**Evidence:**

The check on all these routes is:
```typescript
const userId = request.headers.get("x-user-id")?.trim()
if (!userId) {
  return NextResponse.json({ error: "Unauthorized." }, { status: 401 })
}
```

This only verifies that a non-empty string is present in the header. It performs no database lookup against the session cookie, no token verification, and no signature check. The user ID is a plain numeric auto-increment value (e.g., `1`, `2`, `3`).

The session cookie (`sims-auth`) contains this raw user ID value. The `/api/auth/session` endpoint (which uses proper cookie validation via `requireAuthenticatedUser`) exposes the logged-in user's `id` field to the client JavaScript. Client code then passes `user.id` as the `x-user-id` header on all subsequent requests.

**Proof-of-risk / Reproduction:**

```bash
# Step 1: No login needed if IDs are guessable. Try user ID "1":
curl -X GET https://SIMS_HOST/api/tickets \
  -H "x-user-id: 1"

# Step 2: If the attacker has any session (e.g., a "User" role account),
# they can escalate by forging another user's ID with higher privileges:
curl -X POST "https://SIMS_HOST/api/merchants/import" \
  -H "x-user-id: 2"
```

An external attacker with network access to the server who can enumerate user IDs (they are sequential integers per the auto_increment schema) can read all tickets, all merchant data, trigger a POS API sync, delete/upload files, etc.

The correct controls (`requireAuthenticatedUser`, `getAuthenticatedUser`) are only used in a minority of routes: `/api/auth/session`, `/api/profile`, `/api/users`, and `/api/users/[userId]`.

**Remediation:**

Replace every `request.headers.get("x-user-id")` auth check with `requireAuthenticatedUser(request)` from `src/lib/auth.ts`. This function reads the `sims-auth` cookie, looks up the user in the database, and confirms `status === "active"`. The client should not need to send any `x-user-id` header — the server derives identity from the signed (httpOnly) session cookie.

Remove the `x-user-id` header from all client-side fetches in the `(app)/` pages and from `src/lib/upload-client.ts`.

#### H-1: `/api/uploads/view` File Proxy Endpoint is Completely Unauthenticated

**Risk:** Any person on the internet who knows or can guess/enumerate an object storage key can download any file ever uploaded — including support ticket attachments, avatars, and any PLUS update spreadsheets.

**Affected File:** `/Users/hafiz/sims-io/src/app/api/uploads/view/route.ts` — line 6

**Evidence:**

```typescript
export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url)
  const key = searchParams.get("key")?.trim()
  if (!key) {
    return NextResponse.json({ error: "Missing key." }, { status: 400 })
  }
  // No auth check. Fetches and streams any object from the bucket.
  const result = await getObjectStream(bucket, key)
  ...
}
```

The endpoint is also not included in the middleware matcher and would not be protected even if middleware were extended to cover it. Object keys follow the pattern `{folder}/{userId}/{timestamp}-{random}.{ext}` which provides some obscurity but not security.

The endpoint also sets `Cache-Control: public, max-age=86400` (line 35), which means CDN and browser caches will serve these files to unauthenticated parties.

**Proof-of-risk:**

```bash
curl "https://SIMS_HOST/api/uploads/view?key=support-form/public/1234567890-123456789.pdf"
```

Any guessed or leaked key returns the raw file content with no authentication.

**Remediation:**

Add an authentication check (cookie-based session) at the start of the GET handler. Alternatively, generate short-lived pre-signed MinIO URLs server-side (the AWS S3 SDK supports `GetObjectCommand` with a presigner) and redirect authenticated clients to those, removing the need for this proxy entirely. Remove the `public` directive from `Cache-Control` on authenticated content.

#### H-2: Session Cookie Contains Raw Database User ID (No Session Token)

**Risk:** The `sims-auth` cookie value is the literal database primary key (e.g., `"1"`, `"42"`). If the cookie value leaks (logs, network interception on HTTP, shared environment), an attacker gains the user's persistent identity. Rotating credentials (password reset) does not invalidate existing sessions. There is no way to log out a user remotely by invalidating a server-side session record.

**Affected File:** `/Users/hafiz/sims-io/src/lib/auth.ts` — lines 131-143

**Evidence:**

```typescript
export function setAuthCookie(response: NextResponse, userId: string, remember: boolean) {
  response.cookies.set(SESSION_COOKIE_NAME, userId, {
    httpOnly: true,
    sameSite: "lax",
    secure: process.env.NODE_ENV === "production",
    maxAge: getSessionMaxAge(remember),
    path: "/",
  })
}
```

The cookie is `secure: true` only in production. In development or staging environments without `NODE_ENV=production`, the cookie is sent over plaintext HTTP.

**Remediation:**

Generate a random opaque session token (e.g., `randomBytes(32).toString('hex')`), store it hashed in a `sessions` table along with user ID and expiry, and store only the token in the cookie. On every request, look up the hashed token in the sessions table. This enables server-side logout (delete the session row), revocation after password reset, and ensures the cookie value has no meaning if leaked. Alternatively, adopt a signed JWT approach, but this does not allow server-side revocation.

#### H-3: No Rate Limiting on Login, Password Reset, or Public Form Endpoints

**Risk:** Brute-force credential attacks on `/api/auth/login`, token enumeration on `/api/auth/forgot-password`, and spam/bot abuse of `/api/supportform/submit` and `/api/leads` are unrestricted.

**Affected Files:**
- `/Users/hafiz/sims-io/src/app/api/auth/login/route.ts`
- `/Users/hafiz/sims-io/src/app/api/auth/forgot-password/route.ts`
- `/Users/hafiz/sims-io/src/app/api/supportform/submit/route.ts`
- `/Users/hafiz/sims-io/src/app/api/leads/route.ts`

**Evidence:**

No rate limiting middleware exists anywhere in the codebase (confirmed via grep). The middleware (`middleware.ts`) performs only cookie presence checks with no throttling. The `leads` POST endpoint has optional reCAPTCHA which is disabled when `RECAPTCHA_SECRET_KEY` is unset, providing no bot protection in that case. The support form submit endpoint has no spam protection at all.

The password minimum is only 6 characters (enforced in `/api/auth/activate/route.ts` line 17), meaning brute-force has a large practical search space reduction.

**Remediation:**

- Add rate limiting middleware using a package such as `@upstash/ratelimit` or an in-memory store for single-node deployments. Apply per-IP limits on auth endpoints (e.g., 10 attempts per 15 minutes).
- Add account lockout or exponential backoff after N failed login attempts.
- Make reCAPTCHA mandatory (not optional via env var absence) on all public forms.
- Raise minimum password length to at least 12 characters.

#### H-4: Vulnerable Dependencies — `fast-xml-parser` (Stack Overflow / Entity Expansion)

**Risk:** The AWS SDK dependency chain includes `fast-xml-parser` versions affected by CVE-2026-26278 and related entity expansion bypasses. An attacker who can influence XML content processed by the S3 SDK (e.g., by crafting malicious object metadata responses) could trigger a stack overflow or DoS.

**Affected Package:** `fast-xml-parser` (via `@aws-sdk/xml-builder`) — `npm audit` severity: HIGH

**Evidence:** `npm audit` output reports:
```
fast-xml-parser  4.0.0-beta.3 - 5.5.6
Severity: high
fast-xml-parser has stack overflow in XMLBuilder with preserveOrder
fast-xml-parser affected by numeric entity expansion bypassing all entity expansion limits
```

**Remediation:** Run `npm audit fix`. The `@aws-sdk` packages have patched versions available. Pin `@aws-sdk/xml-builder` to a version above 3.972.8.

#### H-5: Vulnerable Dependency — `xlsx` (SheetJS) — Multiple High-Severity Vulnerabilities

**Risk:** The `xlsx` package (SheetJS) has known high-severity vulnerabilities with no available fix from the vendor. It is used extensively for CSV, XLSX, and PDF export features. Processing attacker-controlled XLSX uploads via the PLUS import feature could exploit these vulnerabilities.

**Affected Files:**
- `/Users/hafiz/sims-io/src/app/api/merchants/export/route.ts`
- `/Users/hafiz/sims-io/src/app/api/plus/update/upload/route.ts` (via `previewPlusTemplate`)
- `/Users/hafiz/sims-io/src/lib/plus-import.ts`

**Evidence:** `npm audit` reports:
```
xlsx
Severity: high (multiple CVEs)
No fix available
```

The PLUS import flow accepts an uploaded XLSX file, then parses it with `xlsx` on the server side (`previewPlusTemplate` in `plus-import.ts`). This is a realistic exploitation path.

**Remediation:**

For server-side XLSX parsing, replace `xlsx` with `exceljs` (actively maintained, no known critical CVEs) or `@fast-csv/parse` for CSV-only cases. For export-only uses (CSV generation), use Node's built-in string manipulation to produce RFC 4180-compliant CSV, eliminating the dependency entirely for that use case.

#### M-1: Cookie Stores Raw User ID — `secure` Flag Off in Non-Production

**Risk:** In development and staging environments (`NODE_ENV !== 'production'`), the `sims-auth` and `sims-google-state` cookies are set without the `Secure` flag, meaning they can be transmitted over HTTP and intercepted.

**Affected Files:**
- `/Users/hafiz/sims-io/src/lib/auth.ts` — line 139
- `/Users/hafiz/sims-io/src/app/api/auth/google/start/route.ts` — line 33

**Evidence:**
```typescript
secure: process.env.NODE_ENV === "production",
```

**Remediation:** The Secure flag should also be set whenever the deployment is accessed via HTTPS, regardless of `NODE_ENV`. Use `process.env.APP_BASE_URL?.startsWith("https://")` or an explicit `SECURE_COOKIES=true` env var as the condition.

#### M-2: No Security Response Headers

**Risk:** The application does not set any HTTP security headers, exposing users to clickjacking, MIME-type sniffing attacks, and making XSS exploitation easier.

**Affected File:** `/Users/hafiz/sims-io/next.config.ts` — no `headers()` function present

**Evidence:** The `next.config.ts` file contains no `headers` export. Verified by grep — no `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, or `Strict-Transport-Security` headers are configured anywhere.

**Remediation:** Add a `headers()` function to `next.config.ts`:
```typescript
async headers() {
  return [{
    source: '/(.*)',
    headers: [
      { key: 'X-Frame-Options', value: 'DENY' },
      { key: 'X-Content-Type-Options', value: 'nosniff' },
      { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
      { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains' },
      // Content-Security-Policy: define after inventorying all external script/style sources
    ]
  }]
}
```

#### M-3: Google OAuth — Error Message Leaked Via Redirect URL Parameter

**Risk:** The Google OAuth callback redirects to `/login?error=<code>` where `<code>` is a plain error code like `sso_failed`, `sso_not_allowed`, `account_inactive`, `activation_required`. These error codes reveal account state to unauthenticated observers (e.g., distinguishing "email exists but inactive" from "email not registered").

**Affected File:** `/Users/hafiz/sims-io/src/app/api/auth/google/callback/route.ts` — lines 40, 134, 137, 139, 144

**Evidence:**
```typescript
// Line 134:
if (!user) { return redirectToLogin(request, "sso_not_allowed") }
// Line 137:
if (user.status === "pending_activation") { return redirectToLogin(request, "activation_required") }
// Line 139:
if (user.status !== "active") { return redirectToLogin(request, "account_inactive") }
```

The error code `account_inactive` reveals that the email address is registered but deactivated. `activation_required` reveals the email is registered but not yet activated.

**Remediation:** Collapse all non-`sso_failed` / `sso_unavailable` error codes to a single generic `sso_not_allowed` code. The distinction between "inactive" and "not registered" is useful only to an attacker for account enumeration.

#### M-4: Plain HTTP Default for POS API Import URL

**Risk:** Credentials (Bearer token retrieved from the authenticated POS API session) are sent over an unencrypted channel when `POS_IMPORT_URL` is not explicitly configured.

**Affected File:** `/Users/hafiz/sims-io/src/lib/pos-api.ts` — line 8

**Evidence:**
```typescript
const DEFAULT_IMPORT_URL = "http://api.getslurp.com/api/franchise-retrieve/"
```

While `DEFAULT_AUTH_URL` uses HTTPS, the import URL defaults to HTTP. The POS API token is attached to all subsequent requests. Any network observer between the server and the API can read the token and use it.

**Remediation:** Change the default to `https://api.getslurp.com/api/franchise-retrieve/`. Add a startup check that warns/errors if `POS_IMPORT_URL` is configured with an HTTP scheme.

#### M-5: CSAT Token Stored and Queried as Plaintext

**Risk:** The CSAT survey token is stored in the database as plaintext (`csat_tokens.token`). If the database is compromised, all active CSAT tokens are immediately usable. The CSAT submission endpoint at `/api/csat/[token]` queries by this plaintext token directly.

**Affected Files:**
- `/Users/hafiz/sims-io/src/app/api/csat/[token]/route.ts` — lines 42-60
- `/Users/hafiz/sims-io/src/app/api/tickets/[ticketId]/route.ts` — lines 614-620
- `/Users/hafiz/sims-io/src/app/api/tickets/[ticketId]/csat/share/route.ts`

**Evidence:**

In `tickets/[ticketId]/route.ts`:
```typescript
generatedCsatToken = randomUUID()
await pool.query(`INSERT INTO csat_tokens (ticket_id, token, expires_at) VALUES (?, ?, ...)`, [ticketId, generatedCsatToken])
```

The token is stored as-is, not hashed. Compare with `auth_tokens` (activation, password_reset) which correctly stores only `token_hash = hashOpaqueToken(token)`.

**Remediation:** Hash CSAT tokens before storing, using the same `hashOpaqueToken` pattern already in `src/lib/auth.ts`. Query by hash at lookup time. The raw token is only needed in the URL sent to the merchant — it never needs to be recoverable from the database.

#### M-6: `/api/supportform/categories` and `/api/supportform/submit` — No Spam/Abuse Controls Beyond Basic Field Validation

**Risk:** The support form submission endpoint has no rate limiting, no CAPTCHA, and no honeypot (unlike the demo form, which at least has an optional CAPTCHA and a `company_site` honeypot field). An attacker can flood the ticket database or MinIO storage.

**Affected Files:**
- `/Users/hafiz/sims-io/src/app/api/supportform/submit/route.ts`
- `/Users/hafiz/sims-io/src/app/api/supportform/categories/route.ts`

**Evidence:** The categories endpoint has no authentication check at all (line 42 — `export async function GET()` with no auth). The submit endpoint has no rate limiting, CAPTCHA, or honeypot.

#### L-2: `dangerouslySetInnerHTML` in Chart Component with Config Values

**Risk:** The chart component injects CSS custom properties from a developer-supplied `ChartConfig` object using `dangerouslySetInnerHTML`. If `ChartConfig` values come from user-controlled data (merchant names, categories, etc.) without sanitization, this could be an XSS vector.

**Affected File:** `/Users/hafiz/sims-io/src/components/ui/chart.tsx` — lines 39-47

**Evidence:**
```typescript
<style dangerouslySetInnerHTML={{ __html: `
${colorConfig.map(([key, cfg]) => {
  return `[data-chart=${id}] { --color-${key}: ${cfg.color}; }`
}).join("\n")}
` }} />
```

Neither `id`, `key`, nor `cfg.color` are sanitized before injection. If a `color` value were set to `red; } body { display:none; } x {`, it would inject arbitrary CSS.

**Assessment:** In the current codebase, `ChartConfig` values are all hardcoded strings in page components (not database-driven), making exploitation unlikely. Labeled low for that reason, but requires manual verification if config ever becomes dynamic.

**Remediation:** Validate that `cfg.color` is a valid CSS color (e.g., matches `/^#[0-9a-fA-F]{3,8}$|^rgb|^hsl/`) before injection. Sanitize `id` and `key` values.

#### L-5: Logout Does Not Invalidate Server-Side Session

**Risk:** Logout only clears the client-side cookie. Because the session is the bare user ID and not a server-stored token, there is no session record to invalidate. A stolen cookie remains valid until its 7-day (or 30-day) TTL expires.

**Affected File:** `/Users/hafiz/sims-io/src/lib/auth.ts` — lines 145-153

**Evidence:**
```typescript
export function clearAuthCookie(response: NextResponse) {
  response.cookies.set(SESSION_COOKIE_NAME, "", { maxAge: 0, ... })
}
```

No database operation is performed. This is a direct consequence of Finding H-2.

**Remediation:** Addressed by implementing server-side session tokens (H-2 remediation). After that, logout should delete the session record from the database.

---

#### L-6: `sims-auth` Cookie `sameSite` is `lax`, Not `strict`

**Risk:** `SameSite=lax` permits the cookie to be sent on top-level navigations from other origins (e.g., `<a href="...">` links). This leaves some CSRF surface for GET endpoints that perform side effects (though in this app, most side effects are on POST routes).

**Affected File:** `/Users/hafiz/sims-io/src/lib/auth.ts` — line 138

**Remediation:** Change `sameSite: "lax"` to `sameSite: "strict"` unless there is a specific cross-site navigation requirement that necessitates lax.

---

#### L-7: No CSRF Tokens for State-Mutating API Routes

**Risk:** While `SameSite=lax` cookies provide partial CSRF protection, there are no explicit CSRF tokens. Routes that accept JSON bodies are protected by the fact that browsers require CORS preflight for non-simple `Content-Type: application/json` requests — but routes accepting `multipart/form-data` (uploads, support form) do not benefit from this protection.

**Affected Files:**
- `/Users/hafiz/sims-io/src/app/api/uploads/route.ts`
- `/Users/hafiz/sims-io/src/app/api/supportform/submit/route.ts`

**Remediation:** Implement `SameSite=strict` cookies (L-6) as the primary protection. For defense-in-depth, add a double-submit CSRF token to mutating form endpoints.
